# Mini DBMS API Server 구름 과제 상세 설계 선택지 문서

## 1. 문서 목적

이 문서는 최종 명세서에서 채택한 선택 외에, 설계 과정에서 고려할 수 있는 다른 옵션들을 정리한 참고 문서다.  
각 항목에 대해 다음을 함께 설명한다.

1. 옵션의 의미
2. 장점
3. 단점
4. 적합한 사용처
5. 이번 과제에서 채택 여부

이 문서를 읽으면 "왜 특정 선택을 했는지"뿐 아니라 "왜 다른 선택을 하지 않았는지"까지 설명할 수 있다.

---

## 2. 통신 프로토콜 선택

### 옵션 A. HTTP/1.1 + JSON

설명:

- 웹 API에서 가장 익숙한 방식이다.
- 요청은 `POST /query`, 응답은 JSON으로 주고받는다.

장점:

- 외부 클라이언트, 브라우저, `curl`, Postman 등에서 바로 사용할 수 있다.
- API 서버라는 과제 목적을 가장 자연스럽게 보여준다.
- 테스트 문서와 데모가 쉽다.

단점:

- C에서 HTTP 요청 파싱 코드를 따로 작성해야 한다.
- 소켓 프로그래밍만 하는 것보다 헤더 처리 부담이 있다.

적합한 사용처:

- 외부 시스템 연동이 중요한 과제
- REST API 또는 웹 API 학습
- 데모와 검증이 중요한 프로젝트

이번 과제의 판단:

- 채택
- 이유: 외부 클라이언트 사용성과 데모 편의성이 가장 크고, 과제 제목 자체가 API 서버이기 때문이다.

### 옵션 B. TCP 기반 커스텀 프로토콜

설명:

- HTTP를 쓰지 않고 소켓으로 직접 문자열을 주고받는 방식이다.

장점:

- 구현이 가장 가볍다.
- HTTP 헤더를 처리하지 않아도 된다.

단점:

- 외부 클라이언트가 바로 사용하기 어렵다.
- API 서버라기보다는 소켓 서버에 가깝게 보일 수 있다.
- 명세 설명과 디버깅 도구 활용성이 떨어진다.

적합한 사용처:

- 내부 전용 통신
- 네트워크 프로토콜 학습
- HTTP가 불필요한 실험용 서버

이번 과제의 판단:

- 미채택
- 이유: 과제 요구사항의 "외부 클라이언트에서 DBMS 기능 사용"을 설명할 때 HTTP보다 설득력이 약하다.

### 옵션 C. HTTP + Plain Text Body

설명:

- HTTP는 사용하지만 요청 본문에 SQL 문자열만 그대로 넣는 방식이다.

장점:

- JSON 파싱 없이 구현할 수 있다.
- 가장 빠르게 만들 수 있다.

단점:

- 나중에 추가 메타데이터를 붙이기 어렵다.
- 입력 검증 결과를 구조적으로 다루기 어렵다.
- 문서화 시 확장성이 부족하다.

적합한 사용처:

- 정말 최소한의 PoC
- 내부 도구 수준의 서버

이번 과제의 판단:

- 미채택
- 이유: 구현은 더 쉽지만, JSON 기반 응답과 잘 맞지 않고 확장성이 떨어진다.

## 3. 엔드포인트 설계 선택

### 옵션 A. `POST /query` 하나로 통합

설명:

- 클라이언트가 SQL 전체를 서버로 보내고 서버가 해석한다.

장점:

- "SQL 요청을 병렬 처리한다"는 과제 요구를 가장 직접적으로 표현한다.
- 기존 SQL 처리기를 자연스럽게 재사용할 수 있다.
- URL 설계가 단순하다.

단점:

- REST 자원 중심 API처럼 보이지는 않는다.
- 잘못된 SQL 문자열 검증이 필요하다.

적합한 사용처:

- DB 게이트웨이
- SQL API 서버
- 기존 SQL 엔진 재사용 프로젝트

이번 과제의 판단:

- 채택

### 옵션 B. `/select`, `/insert` 엔드포인트 분리

설명:

- `POST /insert`, `POST /select`처럼 기능별 엔드포인트를 나누는 방식이다.

장점:

- 클라이언트 입력 검증이 쉬워진다.
- 각 API 문서가 직관적이다.

단점:

- SQL 처리기 재사용성이 떨어진다.
- 과제의 핵심인 "SQL 요청 처리" 느낌이 약해진다.
- API와 내부 SQL 모델이 이중화된다.

적합한 사용처:

- SQL이 아니라 자원 중심 조작이 목표인 서비스
- 내부적으로 ORM 스타일 호출을 원하는 경우

이번 과제의 판단:

- 미채택

### 옵션 C. REST 자원 중심 API

설명:

- `/tables/users/rows` 형태로 자원을 직접 조작하는 방식이다.

장점:

- 일반 웹 서비스 스타일과 유사하다.
- 클라이언트 입장에서 명확한 경우가 있다.

단점:

- 기존 SQL 엔진을 굳이 사용할 이유가 약해진다.
- 과제의 방향과 맞지 않는다.

적합한 사용처:

- SQL 엔진이 없는 CRUD API

이번 과제의 판단:

- 미채택

## 4. 요청 본문 형식 선택

### 옵션 A. JSON `{ "sql": "..." }`

장점:

- 필드 확장성이 좋다.
- 잘못된 JSON, 빈 SQL, 누락 필드를 구분하기 쉽다.
- 향후 `request_id`, `client_tag`, `benchmark_case` 같은 필드를 붙이기 좋다.

단점:

- 최소한의 JSON 파싱이 필요하다.

적합한 사용처:

- 구조화된 API
- 향후 확장 가능성을 열어 두고 싶은 서버

이번 과제의 판단:

- 채택

### 옵션 B. 본문 전체를 SQL 문자열로 해석

장점:

- 구현이 가장 단순하다.

단점:

- 확장이 어렵다.
- 입력 오류를 세밀하게 분류하기 어렵다.

적합한 사용처:

- 짧은 데모
- 일회성 실험 서버

이번 과제의 판단:

- 미채택

### 옵션 C. `x-www-form-urlencoded`

장점:

- HTML 폼과 호환된다.

단점:

- DB API 서버 맥락에서는 특별한 이점이 없다.
- JSON보다 표현력이 제한적이다.

적합한 사용처:

- 브라우저 폼 전송 위주 서비스

이번 과제의 판단:

- 미채택

---

## 5. SQL 범위 선택

### 옵션 A. `INSERT`, `SELECT`, `WHERE =`만 지원

장점:

- parser, executor, 테스트 범위를 통제하기 쉽다.
- API 서버와 동시성 설계에 집중할 수 있다.
- 기존 코드 재사용성이 높다.

단점:

- 기능이 제한적이다.
- 완전한 DBMS처럼 보이진 않는다.

적합한 사용처:

- 학습용 최소 구현
- 아키텍처와 동시성 검증 중심 과제

이번 과제의 판단:

- 채택

### 옵션 B. `UPDATE`, `DELETE`까지 포함

장점:

- CRUD 구색이 맞아 보인다.
- 데모에서 기능 폭이 넓어진다.

단점:

- write 경로가 늘면서 락 정책과 테스트 범위가 커진다.
- 기존 parser와 executor 확장이 필요하다.

적합한 사용처:

- 과제 시간이 충분한 팀
- 기능 완성도를 더 중요하게 보는 프로젝트

이번 과제의 판단:

- 미채택

### 옵션 C. `CREATE TABLE` 외부 지원

장점:

- 외부 클라이언트가 스키마를 직접 만들 수 있다.
- DBMS다운 느낌이 강해진다.

단점:

- parser, schema 저장, runtime 갱신까지 모두 손봐야 한다.
- 초기 구현 리스크가 크다.

적합한 사용처:

- DBMS 자체 구현을 더 중시하는 프로젝트

이번 과제의 판단:

- 미채택

### 옵션 D. 다중 SQL 요청 허용

설명:

- 하나의 `POST /query`에 `INSERT ...; SELECT ...;`처럼 여러 문장을 보내는 방식이다.

장점:

- 기존 parser는 이론상 다중 문장 분해가 가능하다.
- 배치 처리처럼 보일 수 있다.

단점:

- 락 획득 정책과 트랜잭션 의미가 애매해진다.
- 요청 실패 시 어느 문장까지 성공했는지 정의해야 한다.
- 성능 측정 기준이 흐려진다.

적합한 사용처:

- 배치 실행기
- CLI 스타일 SQL 러너

이번 과제의 판단:

- 미채택
- 이유: API 서버 요청 모델은 단일 문장이 더 단순하고 분석하기 쉽다.

## 6. 스키마 정의 방식 선택

### 옵션 A. `.schema` 파일, 컬럼명만 저장

장점:

- 현재 코드베이스와 바로 맞는다.
- 구현 범위가 작다.
- 로딩 규칙이 명확하다.

단점:

- 타입 정보가 없다.
- 제약 조건을 표현하기 어렵다.

적합한 사용처:

- 기존 파일 기반 엔진 재사용
- 최소 구현

이번 과제의 판단:

- 채택

### 옵션 B. `schema.sql` 사용

장점:

- DBMS다운 느낌이 강하다.
- `CREATE TABLE` 문법 기반 설명이 자연스럽다.

단점:

- SQL 파서 확장이 필요하다.
- 현재 코드와 직접 맞지 않는다.

적합한 사용처:

- SQL 파서 확장까지 할 여유가 있는 프로젝트

이번 과제의 판단:

- 미채택

### 옵션 C. JSON 스키마 파일 사용

장점:

- 타입, 제약 조건, 메타데이터 표현이 쉽다.

단점:

- C에서 JSON 파서 의존성이 생기거나 직접 파싱해야 한다.
- 기존 SQL 처리기와 스키마 경로가 분리된다.

적합한 사용처:

- 메타데이터가 많은 시스템
- 스키마 진화가 중요한 프로젝트

이번 과제의 판단:

- 미채택

### 옵션 D. 코드에 테이블 구조 하드코딩

장점:

- 가장 빠르게 만들 수 있다.

단점:

- 확장성과 설명 가능성이 매우 낮다.
- 과제 결과물 품질이 떨어진다.

적합한 사용처:

- 하루짜리 데모

이번 과제의 판단:

- 미채택

## 7. 타입 시스템 선택

### 옵션 A. 타입 시스템 제외

장점:

- 현재 구조와 잘 맞는다.
- 응답과 저장 구조를 단순하게 유지할 수 있다.

단점:

- `INT`, `TEXT` 같은 검증이 없다.
- 잘못된 값 삽입을 막기 어렵다.

적합한 사용처:

- 최소 구현
- 구조 검증 중심 프로젝트

이번 과제의 판단:

- 채택

### 옵션 B. `INT`, `TEXT` 두 타입만 지원

장점:

- 설명하기 쉽다.
- 결과와 입력 검증이 더 자연스럽다.

단점:

- `.schema` 형식 변경과 executor 검증 로직 추가가 필요하다.
- 현재 코드와 직접 맞지 않는다.

적합한 사용처:

- 약간의 추가 구현 여유가 있는 팀

이번 과제의 판단:

- 미채택

### 옵션 C. 더 풍부한 타입 체계

장점:

- DBMS다운 느낌이 강해진다.

단점:

- 최소 구현 범위를 크게 초과한다.

적합한 사용처:

- 장기 프로젝트

이번 과제의 판단:

- 미채택

## 8. 동시성 모델 선택

### 옵션 A. 고정 크기 스레드 풀

장점:

- 스레드 수가 통제된다.
- 성능 비교 실험에 적합하다.
- 과제 요구사항과 직접 일치한다.

단점:

- 큐 포화 시 정책을 설계해야 한다.

적합한 사용처:

- 요청 단위 병렬 처리 서버
- 교육용 멀티스레드 과제

이번 과제의 판단:

- 채택

### 옵션 B. 요청마다 스레드 생성

장점:

- 개념상 이해가 쉽다.
- 큐가 없어도 된다.

단점:

- 요청이 많아지면 스레드 생성/소멸 비용이 크다.
- 자원 통제가 어렵다.

적합한 사용처:

- 요청 수가 매우 적은 서버

이번 과제의 판단:

- 미채택

### 옵션 C. 단일 스레드 이벤트 루프

장점:

- 락 문제가 거의 없다.
- 구현이 일정 부분 단순해질 수 있다.

단점:

- 과제 요구사항인 스레드 풀과 맞지 않는다.
- CPU 바운드 SQL 실행 병렬성이 없다.

적합한 사용처:

- I/O 위주 서버

이번 과제의 판단:

- 미채택

## 9. 락 정책 선택

### 옵션 A. 전역 `pthread_rwlock_t`

장점:

- `SELECT` 병렬 실행이 가능하다.
- write 경로는 안전하게 직렬화된다.
- 구현과 설명의 균형이 좋다.

단점:

- write-heavy workload에서 병목이 생긴다.
- 테이블이 여러 개여도 락이 하나라 병렬성이 제한된다.

적합한 사용처:

- 최소 구현
- 안정성을 우선하는 학습용 서버

이번 과제의 판단:

- 채택

### 옵션 B. 전역 `mutex`

장점:

- 구현이 가장 단순하다.

단점:

- `SELECT`끼리도 병렬성이 없다.
- 스레드 풀의 장점을 살리기 어렵다.

적합한 사용처:

- 매우 짧은 데모
- 동시 읽기 이점이 중요하지 않은 경우

이번 과제의 판단:

- 미채택

### 옵션 C. 테이블 단위 `RWLock`

장점:

- 서로 다른 테이블은 더 잘 병렬화된다.

단점:

- 테이블 관리 로직이 복잡해진다.
- 런타임 캐시와 락 수명 관리가 필요하다.

적합한 사용처:

- 다중 테이블 workload
- 확장형 구조

이번 과제의 판단:

- 미채택

### 옵션 D. 저장소/인덱스 세분 락

장점:

- 이론상 최고 성능을 노릴 수 있다.

단점:

- 데드락과 정합성 설계가 어렵다.
- 과제 범위를 넘는다.

적합한 사용처:

- 장기적인 DB 엔진 개발

이번 과제의 판단:

## 17. 이번 과제에서 가장 중요한 설명 포인트

이 프로젝트를 발표하거나 문서로 설명할 때는 다음 논리로 정리하면 된다.

1. 외부 인터페이스는 HTTP/JSON으로 단순하고 명확하게 만들었다.
2. 내부 실행은 기존 SQL 처리기와 B+ 트리를 재사용했다.
3. 요청 처리는 고정 크기 스레드 풀과 bounded queue로 통제했다.
4. 동시성은 전역 `RWLock`으로 단순화하여 안정성을 우선했다.
5. 대신 성능 측정을 필수 검증으로 넣어 이 구조의 한계까지 수치로 보여주기로 했다.

이 다섯 줄만 분명하면, 왜 이 설계가 최소 구현 기준에서 합리적인지 충분히 설명할 수 있다.
