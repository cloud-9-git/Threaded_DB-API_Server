# 스레드, 스레드풀, 동기/비동기, 블로킹/논블로킹, 락, 처리량 정리

## 문서 목적

이 문서는 지금까지의 대화 내용을 빠짐없이 정리하되, 중복만 제거하고, 이해하기 쉬운 흐름으로 다시 재구성한 정리본이다.

핵심은 이거다.

- `싱글 스레드 / 멀티 스레드 / thread-per-request / 스레드풀`은 **작업자를 어떻게 운영하느냐**에 대한 이야기다.
- `동기 / 비동기`는 **작업이 끝날 때까지 어떻게 기다리느냐**에 대한 이야기다.
- `블로킹 / 논블로킹`은 **기다리는 동안 멈추느냐, 안 멈추느냐**에 대한 이야기다.
- `동시성 / 병렬성`은 **여러 일을 겹쳐 다루느냐**, **진짜 동시에 실행하느냐**에 대한 이야기다.

즉, 이 용어들은 전부 같은 종류가 아니라 서로 다른 축의 개념이다. 그래서 헷갈리기 쉽다.

예를 들어:

- `싱글 스레드 + 비동기`도 가능하다.
- `멀티 스레드 + 동기`도 가능하다.

즉, `싱글 스레드`와 `비동기`는 서로 반대말이 아니다.

---

## 1. 먼저 큰 그림부터

처음 헷갈리는 이유는 이 용어들이 전부 같은 질문에 답하는 게 아니기 때문이다.

| 축 | 이 질문에 답한다 | 대표 용어 |
|---|---|---|
| 작업자 구조 | 누가 몇 개의 일을 처리하나 | `싱글 스레드`, `멀티 스레드`, `thread-per-request`, `스레드풀` |
| 기다리는 방식 | 작업 완료를 어떻게 기다리나 | `동기`, `비동기` |
| 멈춤 여부 | 기다리는 동안 멈추나 | `블로킹`, `논블로킹` |
| 실행 방식 | 여러 일을 어떻게 다루나 | `동시성`, `병렬성` |
| 공유 자원 보호 | 같이 쓰는 데이터를 어떻게 지키나 | `mutex`, `rwlock`, `semaphore`, 락 범위 |

이 축을 분리해서 보지 않으면 이런 오해가 생긴다.

- `비동기 = 멀티 스레드`
- `I/O 서버 = 무조건 싱글 스레드가 좋음`
- `스레드 수만 늘리면 빨라짐`

이 문서의 목적은 이런 오해를 한 번에 정리하는 것이다.

## 2. 작업자 구조: 누가 일을 처리하나

### 싱글 스레드

- 작업자 1명으로 처리하는 방식이다.
- 한 순간에 한 흐름만 처리한다고 보면 된다.
- 구조가 단순하고 디버깅이 쉽다.
- 동기화 문제나 공유 자원 충돌이 상대적으로 적다.
- 하지만 하나가 오래 걸리면 뒤 요청들이 줄줄이 기다리기 쉽다.

쉽게 말하면:

- 싱글 스레드 = 직원 1명

### 멀티 스레드

- 작업자 여러 명이 일을 나눠 처리하는 방식이다.
- 여러 요청을 동시에 또는 겹쳐 처리할 수 있다.
- 전체 처리량이 늘어날 수 있다.
- 하지만 락, 데드락, 경쟁 조건, 컨텍스트 스위칭 같은 복잡성이 생긴다.
- 구현이 어려워질 수 있고, 상황에 따라 오히려 느려질 수도 있다.

쉽게 말하면:

- 멀티 스레드 = 직원 여러 명

### thread-per-request

- 요청 1개가 들어올 때마다 스레드 1개를 새로 만드는 방식이다.
- 흐름은 보통 `요청 도착 -> 새 스레드 생성 -> 요청 처리 -> 스레드 종료`다.
- 구조는 직관적이고 이해하기 쉽다.
- 하지만 요청이 많아지면 스레드 수가 같이 폭증할 수 있다.
- 스레드 생성/종료 비용, 메모리 사용량, 컨텍스트 스위칭 비용이 커질 수 있다.
- 그래서 작은 규모나 학습용에서는 단순하지만, 실서비스에서는 `thread pool`보다 불리한 경우가 많다.

쉽게 말하면:

- thread-per-request = 손님 올 때마다 직원 한 명 새로 뽑는 방식

### 스레드풀

- 요청이 들어올 때마다 새 스레드를 만드는 대신, 미리 만들어 둔 스레드들을 재사용하는 방식이다.
- 멀티 스레드를 더 안정적이고 효율적으로 굴리기 위한 운영 방식이라고 보면 된다.
- 실무에서 `스레드풀`이라고 하면 보통 `멀티스레드 방식`이 맞다.
- 스레드 생성/파괴 비용을 줄일 수 있다.
- 동시에 일하는 개수를 제한해서 서버 폭주를 막는 데 도움이 된다.
- 다만 풀 크기 조절이 중요하다.
- 너무 작으면 대기열이 길어진다.
- 너무 크면 CPU/메모리 낭비가 생긴다.
- 풀 크기를 1로 두면 기술적으로는 스레드풀처럼 운영해도, 동작은 거의 `싱글 워커`에 가깝다.

쉽게 말하면:

- 스레드풀 = 직원 여러 명을 미리 고용해두고 돌아가며 일 시키는 방식

### 작업자 구조를 비교하면

| 방식 | 어떻게 일하나 | 장점 | 단점 | 잘 맞는 상황 |
|---|---|---|---|---|
| `싱글 스레드` | 작업자 1명이 처리 | 단순하고 디버깅 쉬움 | 하나가 오래 걸리면 뒤 요청이 밀림 | 단순한 프로그램, 동시 요청 거의 없음 |
| `멀티 스레드` | 여러 작업자가 나눠 처리 | 여러 요청을 겹쳐 처리 가능 | 락, 경쟁 조건, 데드락, 컨텍스트 스위칭이 생김 | 동시 요청 처리 필요 |
| `thread-per-request` | 요청마다 새 스레드 생성 | 구조 직관적 | 요청 많아지면 스레드 수 폭증 | 학습용, 소규모 서버 |
| `스레드풀` | 미리 만든 worker 재사용 | 생성 비용 절약, 안정적 운영 | 풀 크기 튜닝 필요 | blocking I/O 기반 실서비스 서버 |

## 3. 기다리는 방식: 끝날 때까지 어떻게 버티나

### 동기(sync)

- 어떤 작업이 끝날 때까지 기다린 뒤 다음으로 넘어가는 방식이다.
- DB나 파일 응답이 올 때까지 그 자리에서 기다린다고 보면 된다.

예:

- `DB 응답 올 때까지 여기서 기다림`

### 비동기(async)

- 작업을 맡겨두고, 끝날 때까지 다른 일을 먼저 하는 방식이다.
- 나중에 결과가 오면 이어서 처리한다.

예:

- `DB야 답 오면 알려줘, 난 그동안 다른 요청 볼게`

쉽게 말하면:

- 동기 = 끝날 때까지 기다림
- 비동기 = 기다리는 동안 다른 일 함

### `await`는 무슨 뜻인가

예를 들어 `await db.query()`라고 쓰면:

- 코드상으로는 그 함수가 잠깐 멈춘 것처럼 보인다.
- 하지만 서버 전체가 멈추는 뜻은 아니다.
- 다른 작업을 하러 갈 수 있다.
- DB 응답이 오면 그때 다시 이어서 실행한다.

즉, `await`는 "여기서 영원히 붙잡혀 있음"이 아니라, "결과가 오면 이어서 처리할 예약을 걸어두는 것"에 가깝다.

### 비동기라고 해서 멀티 스레드라는 뜻은 아니다

- `비동기 = 멀티 스레드`는 아니다.
- 비동기는 `싱글 스레드`로도 가능하다.
- 이유는 CPU가 계속 계산하는 게 아니라, 많은 시간 동안 DB, 파일, 네트워크 응답을 기다리기 때문이다.
- 그 기다리는 시간에 다른 일을 끼워 넣는 것이 비동기의 핵심이다.

### 동기와 비동기를 비교하면

| 개념 | 뜻 | 장점 | 단점 | 잘 맞는 상황 |
|---|---|---|---|---|
| `동기(sync)` | 끝날 때까지 기다린 뒤 다음으로 감 | 흐름이 직관적 | 기다리는 동안 놀기 쉬움 | 단순한 흐름, blocking 라이브러리 |
| `비동기(async)` | 맡겨두고 다른 일을 먼저 함 | 기다리는 시간 동안 다른 요청 처리 가능 | async 스택 필요, 코드 흐름이 익숙해지기 어렵고 blocking 코드가 끼면 효과가 깨짐 | DB/API/파일 I/O가 많은 서버 |

## 4. 블로킹 / 논블로킹: 기다리는 동안 멈추나

### 블로킹(blocking)

- 기다리는 동안 그 작업자가 멈춘다.
- DB, 파일, 네트워크 응답이 올 때까지 손을 놓고 서 있는 느낌이다.

### 논블로킹(non-blocking)

- 기다리는 동안 멈추지 않고 다른 일을 할 수 있다.

입문 단계에서는 보통 이렇게 이해해도 된다.

- `동기 ~= 블로킹`
- `비동기 ~= 논블로킹`

정확히 말하면 완전히 같은 말은 아니지만, 처음 이해할 때는 거의 이렇게 묶어서 생각해도 큰 무리는 없다.

| 개념 | 뜻 | 느낌 |
|---|---|---|
| `블로킹` | 기다리는 동안 그 작업자가 멈춤 | 손을 놓고 응답을 기다림 |
| `논블로킹` | 기다리는 동안 멈추지 않음 | 응답 기다리는 동안 다른 요청 처리 가능 |

## 5. 동시성 / 병렬성: 여러 일을 어떻게 다루나

### 동시성(concurrency)

- 여러 일을 겹쳐서 다루는 개념이다.
- 실제로 한순간에 완전히 동시에 실행되는지가 핵심은 아니다.
- 중요한 건 "안 막히게 운영하는 능력"이다.

예:

- 요청 A가 DB 응답을 기다리는 동안 요청 B를 처리하는 것

### 병렬성(parallelism)

- 여러 일을 진짜 동시에 실행하는 개념이다.
- 여러 CPU 코어를 사용해 동시에 계산을 돌리는 느낌에 가깝다.

예:

- CPU 코어 4개로 계산 4개를 동시에 돌리는 것

쉽게 말하면:

- 동시성 = 여러 일을 안 막히게 겹쳐 처리하는 운영 능력
- 병렬성 = 여러 일을 물리적으로 동시에 실행하는 능력

| 개념 | 뜻 | 핵심 |
|---|---|---|
| `동시성` | 여러 일을 겹쳐서 다룸 | 안 막히게 운영하는 능력 |
| `병렬성` | 여러 일을 진짜 동시에 실행 | 실제로 동시에 계산하는 능력 |

## 6. 가장 많이 헷갈리는 조합

이제부터는 개념을 하나씩 따로 보는 것보다, 실제 조합으로 보는 게 더 이해가 쉽다.

| 조합 | 무슨 뜻인가 | 장점 | 단점 | 잘 맞는 상황 |
|---|---|---|---|---|
| `싱글 스레드 + 블로킹` | 작업자 1명이 기다리는 동안 멈춤 | 단순함 | 전체가 막히기 쉬움 | 아주 단순한 프로그램 |
| `싱글 스레드 + 비동기` | 작업자 1명이 기다리는 동안 다른 요청 처리 | I/O 서버에 강함 | blocking 코드가 끼면 전체 흐름 깨짐 | async 스택이 잘 갖춰진 서버 |
| `멀티 스레드 + 블로킹` | 한 스레드가 기다려도 다른 스레드가 일함 | blocking 라이브러리와 궁합이 좋음 | 락 설계가 중요 | 기존 blocking 코드 기반 서버 |
| `스레드풀 + 블로킹` | 미리 만든 worker들이 요청을 나눠 처리 | 가장 현실적인 운영 방식 | 풀 크기와 락 경합에 민감 | 실서비스 API 서버 |
| `멀티 스레드 + 비동기` | 여러 작업자도 있고 기다리는 동안도 안 멈춤 | 높은 잠재 성능 | 구조가 복잡함 | 특별히 필요한 경우 |

## 7. 예시로 이해하기

### 요청 A와 B 예시

상황:

- 요청 A가 DB 응답을 200ms 기다리는 중이다.
- 그 사이 요청 B가 들어온다.

이때:

- `싱글 스레드 + 블로킹`이면 A를 기다리는 동안 B도 멈출 수 있다.
- `싱글 스레드 + 비동기`면 A는 기다리게 두고 B를 먼저 처리할 수 있다.
- `멀티 스레드`면 다른 스레드가 B를 처리할 수 있다.

이 예시에서 중요한 건, 느린 이유가 "계산이 느려서"가 아니라 "기다리는 동안 놀고 있어서"일 수 있다는 점이다.

### 요청 A가 2초 걸릴 때

- `싱글 스레드`에서 요청 A가 2초 걸리면, 요청 B는 그 뒤에 2초를 더 기다릴 수 있다.
- `멀티 스레드`에서는 A가 DB를 기다리는 동안 B를 다른 스레드가 먼저 처리할 수 있다.

즉, 병렬 처리나 동시 처리의 이점은 주로 "뒤 요청이 덜 막히는 것"에서 크게 나타난다.

## 8. 성능을 볼 때 같이 봐야 하는 것

### 언제 더 빨라지는가

| 상황 | 왜 빨라질 수 있나 | 보통 무엇이 중요하나 |
|---|---|---|
| `CPU 작업이 많을 때` | 여러 코어를 활용하면 실제 계산 속도가 빨라질 수 있음 | `병렬성` |
| `I/O 대기가 많을 때` | 기다리는 시간을 겹쳐 처리할 수 있음 | `동시성` |

예:

- 이미지 처리, 압축, 계산처럼 CPU를 많이 쓰는 작업은 여러 코어를 활용하면 실제 계산 속도가 빨라질 수 있다.
- DB 조회, 파일 읽기, 외부 API 호출 같은 작업에서는 "계산"이 빨라진다기보다 "기다리는 시간을 겹쳐 처리"해서 성능이 좋아진다.

즉:

- CPU 서버에서는 병렬성이 중요하다.
- I/O 서버에서는 동시성이 중요하다.

### 언제 안 빨라질 수 있나

- 작업이 아주 짧고 단순할 때
- 스레드끼리 같은 자원을 많이 공유해서 락 경합이 심할 때
- 코어 수보다 너무 많은 스레드를 돌릴 때
- 스레드 생성 비용이 실제 작업보다 클 때

즉, `스레드 수를 늘린다 = 무조건 빨라진다`는 말은 틀릴 수 있다.

### 처리량(throughput)이란 무엇인가

- `처리량(throughput)`은 단위 시간당 얼마나 많은 요청을 처리하느냐를 뜻한다.
- 예를 들어 초당 10개 요청을 처리하면 `10 req/s`, 초당 100개를 처리하면 `100 req/s`라고 볼 수 있다.
- 처리량은 서버 전체가 얼마나 많이 처리하느냐를 보는 지표다.

함께 자주 나오는 개념은 `응답시간(latency)`이다.

- `처리량`: 서버가 전체적으로 얼마나 많이 처리하느냐
- `응답시간`: 개별 요청 1개가 얼마나 빨리 끝나느냐

그래서 `처리량이 많다 = 각 요청 속도가 무조건 빠르다`는 뜻은 아니다.

예를 들면:

- 요청 1개 처리 자체는 여전히 `100ms`일 수 있다.
- 하지만 동시에 더 많은 요청을 겹쳐 처리할 수 있게 되면 서버 전체는 초당 더 많은 요청을 처리할 수 있다.

즉:

- `처리량 증가`는 서버 전체 성능이 좋아졌다는 뜻에 가깝다.
- `응답시간 감소`는 각 유저 체감 속도가 빨라졌다는 뜻에 가깝다.

둘은 관련은 있지만 같은 말은 아니다.

### "그냥 각각 유저의 속도가 빨라지는 거임?"

중요한 포인트는, 병렬 처리나 동시 처리가 항상 "한 유저의 요청 자체를 엄청 짧게 만든다"는 뜻은 아니라는 점이다.

더 자주 나타나는 효과는 다음과 같다.

- 여러 유저가 동시에 들어와도 덜 막힌다.
- 대기열이 덜 쌓인다.
- 서버 전체 처리량이 올라간다.
- 혼잡할 때 전체 응답성이 좋아진다.

즉:

- `개별 요청 지연시간(latency)`이 줄 수도 있다.
- 하지만 더 흔한 효과는 `동시 접속자가 많을 때 전체 대기시간 감소`와 `처리량 증가`다.

I/O 서버에서는 특히 그렇다.

- "한 요청의 순수 계산 시간"이 엄청 줄어드는 게 아니라,
- "기다리는 동안 다른 요청을 처리해서 덜 막히게 만드는 것"이 핵심이다.

## 9. 락은 왜 성능의 핵심인가

### 락이 성능에 미치는 영향

- 멀티스레드 성능은 단순히 `스레드 개수`만으로 정해지지 않는다.
- 여러 스레드가 같은 자원을 쓰면 락을 잡고 기다리게 된다.
- 기다리는 스레드가 많아질수록 `락 경합(lock contention)`이 생긴다.
- 락 경합이 심하면 병렬처리 이득이 줄고, 오히려 컨텍스트 스위칭 때문에 더 느려질 수도 있다.
- 그래서 멀티스레드 성능은 `공유 자원 + 락 설계`에 크게 좌우된다.

### 자주 나오는 락 종류

| 락 종류 | 역할 | 잘 맞는 상황 | 주의점 |
|---|---|---|---|
| `mutex` | 가장 일반적인 상호배제 | 짧은 공유 메모리 보호 | 범위가 커지면 병목이 커짐 |
| `spinlock` | 잠들지 않고 잠깐 바쁘게 기다림 | 매우 짧은 대기 | 오래 걸리면 CPU만 태움 |
| `rwlock` | 읽기는 같이, 쓰기는 단독 | 읽기는 많고 쓰기는 적을 때 | 쓰기 starvation이나 설계 복잡도에 주의 |
| `semaphore` | 동시에 몇 개까지 허용할지 제한 | 상호배제보다 개수 제한이 필요할 때 | mutex와 목적이 다름 |

### 락 범위가 왜 중요한가

락 범위는 보통 두 가지를 같이 본다.

- `데이터 범위`: 전역 락, 테이블 락, 객체 락, 레코드 락처럼 무엇을 한 번에 묶어 잠그는가
- `코드 범위`: 락을 얼마나 긴 구간 동안 잡고 있는가

예를 들면:

- `전역 락 1개`로 모든 요청을 보호하면 구현은 쉽지만 거의 직렬처리처럼 될 수 있다.
- `테이블별`이나 `레코드별`로 락을 쪼개면 동시성은 좋아질 수 있지만 코드가 복잡해진다.
- 특히 `락을 잡은 채로 DB 호출, 파일 I/O, 네트워크 I/O`까지 해버리면 성능상 매우 불리하다.

실무 감각으로 보면:

- 처음에는 조금 큰 락으로 안전하게 시작할 수 있다.
- 병목이 확인되면 더 작은 범위의 락으로 쪼개는 방향을 검토한다.
- `critical section`은 최대한 짧게 잡는 것이 좋다.

## 10. 우리 서버(DB API 서버) 기준으로 보면

### 서버 성격부터 보면

우리가 이야기한 서버는 보통 `빡센 계산 서버`라기보다 `I/O 위주 DB API 서버`에 가깝다.

즉, 이 서버에서 중요한 건:

- CPU를 더 세게 병렬화하는 것보다
- DB 응답, 파일 읽기, 네트워크 응답을 기다리는 동안 얼마나 안 막히게 처리하느냐이다.

그래서 "I/O 위주 서버면 싱글 스레드가 더 낫다는 뜻인가?"라는 질문에 대한 정확한 답은 다음과 같다.

- `I/O 서버니까 싱글 스레드가 낫다`는 말은 정확하지 않다.
- 더 정확한 말은 `I/O 서버니까 CPU 병렬화보다 동시성 처리가 중요하다`이다.

### 우리 서버에 맞는 선택지

| 조건 | 보통 더 잘 맞는 선택 |
|---|---|
| DB 드라이버와 라이브러리가 `async`를 잘 지원함 | `싱글 스레드 + 비동기` |
| DB 드라이버나 코드가 `blocking` 중심임 | `스레드풀` |
| `싱글 스레드 + 블로킹`만 사용함 | 동시 요청이 늘면 대체로 불리 |

또한 이 서버에서 느린 이유는 보통:

- 계산이 느려서라기보다
- 기다리는 동안 놀고 있어서인 경우가 많다.

### 지금 우리 서버의 락 범위

현재 프로젝트 코드 기준으로 핵심 락 범위는 사실상 `DB 전체`, 더 정확히는 `서버가 공유하는 단일 DbApi / ExecutionContext 전체` 단위라고 볼 수 있다.

이유는 다음과 같다.

- `DbApi` 안에 `pthread_rwlock_t lock`가 하나만 있다.
- `Server`도 `DbApi`를 하나만 공유한다.
- 따라서 지금은 `테이블별 락`, `행(row)별 락`, `인덱스별 락`이 아니라 `공유 DB 실행 컨텍스트 전체를 감싸는 coarse-grained rwlock 1개` 구조다.

현재 동작은 대략 이렇다.

- `SELECT`: 먼저 `write lock`으로 테이블 runtime preload를 수행하고, 그 다음 lock을 풀고 다시 `read lock`을 잡아서 실제 SELECT를 실행한다.
- `INSERT`: `write lock`을 잡은 채로 `execute_statement()` 전체를 수행한다.

이 의미는 다음과 같다.

- `read-read`는 동시에 가능할 수 있다.
- 다만 SELECT의 preload 구간은 `write lock`이라 잠깐 직렬화된다.
- `read-write`는 동시에 진행되지 않는다.
- `write-write`도 동시에 진행되지 않는다.
- 심지어 서로 다른 테이블을 건드리는 요청이어도 같은 lock 하나를 공유하기 때문에 서로 막힐 수 있다.

즉, 지금 구조는 요청별 락이나 테이블별 락보다 훨씬 큰 범위의 락을 쓰는 편이다.

참고로 DB 락 말고도 작은 락은 따로 있다.

- 작업 큐 보호용 `mutex` 1개
- `thread-per-request` 모드의 활성 요청 수를 보호하는 `mutex` 1개

하지만 성능에 가장 큰 영향을 주는 것은 이 작은 보조 락들보다 `DbApi` 안의 전역 `rwlock`이다.

## 11. 실무에서 어떻게 고르면 되나

| 우선순위 | 더 잘 맞는 선택 |
|---|---|
| `단순함`이 가장 중요 | `싱글 스레드` |
| `요청마다 스레드를 새로 만드는 단순한 구조`가 필요 | `thread-per-request` |
| `I/O 대기`를 많이 겹쳐 처리하고 싶음 | `비동기` |
| `blocking 코드`를 버리지 못하면서 동시 처리하고 싶음 | `스레드풀` |
| `CPU 계산`이 진짜 많음 | `멀티 스레드 / 멀티 프로세스` 검토 |

한 줄로 요약하면:

- `thread-per-request`는 단순하지만 잘 안 버티는 편이다.
- `스레드풀`은 실무형 blocking 서버의 기본 선택지다.
- `비동기`는 I/O 서버에서 가장 효율적일 수 있지만 스택이 잘 맞아야 한다.

## 12. 최종 한 줄 요약

- `싱글/멀티 스레드`는 작업자 수
- `thread-per-request`는 요청마다 스레드를 새로 만드는 방식
- `동기/비동기`는 기다리는 방식
- `블로킹/논블로킹`은 기다릴 때 멈추는지 여부
- `동시성`은 안 막히게 겹쳐 처리하는 개념
- `병렬성`은 진짜 동시에 실행하는 개념
- `스레드풀`은 멀티 스레드를 실무적으로 굴리는 방식
- `처리량`은 단위 시간당 처리 가능한 요청 수이고, `응답시간`과는 다른 개념이다
- 멀티스레드 성능은 `스레드 수`뿐 아니라 `락 종류`, `락 범위`, `락을 잡는 시간`에도 크게 좌우된다
- `I/O 서버`에서는 보통 `비동기`나 `스레드풀`이 유리하고, `싱글 스레드 + 블로킹`은 불리한 경우가 많다
