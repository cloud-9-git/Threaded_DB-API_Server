# MiniDB API Server 최소 구현 명세서

## 1. 프로젝트 개요

본 프로젝트는 기존에 구현한 **SQL 처리기**와 **B+ Tree 인덱스**를 활용하여, 외부 클라이언트가 API를 통해 간단한 DBMS 기능을 사용할 수 있도록 하는 **C 기반 미니 DBMS API 서버**이다.

최소 구현에서는 복잡한 REST API나 완전한 SQL 문법을 구현하지 않고, 다음 세 가지에 집중한다.

```text
1. HTTP 요청을 받을 수 있는 API 서버
2. Thread Pool을 통한 요청 처리 구조
3. 기존 SQL 처리기와 B+ Tree 인덱스 연동
```

---

## 2. 최소 구현 목표

### 핵심 목표

* HTTP 서버를 실행한다.
* 클라이언트가 SQL 문자열을 서버로 보낼 수 있다.
* 서버는 요청을 Thread Pool에 전달한다.
* Worker Thread가 SQL을 실행한다.
* SQL 실행 결과를 클라이언트에게 반환한다.
* 공유 자원은 Mutex로 보호한다.
* 기본 테스트와 데모가 가능해야 한다.

## 11. Thread Pool 명세

## 11.1 목적

요청마다 스레드를 새로 생성하지 않고, 미리 생성한 Worker Thread들이 요청을 처리하도록 한다.

---

## 11.2 Thread Pool 구조

```text
Main Thread
  |
  | accept client
  v
Task Queue
  |
  | dequeue
  v
Worker Thread 1
Worker Thread 2
Worker Thread 3
Worker Thread 4
```

---

## 11.3 최소 동작 방식

1. 서버 시작 시 Worker Thread 4개를 생성한다.
2. Main Thread는 `accept()`로 클라이언트 연결을 받는다.
3. 연결된 Socket FD를 Task Queue에 넣는다.
4. Worker Thread는 Queue에서 Socket FD를 꺼낸다.
5. Worker Thread는 HTTP 요청에서 SQL Body를 추출한다.
6. `db_execute(sql)`을 호출한다.
7. 실행 결과를 HTTP Response로 보낸다.
8. Socket을 닫는다.
9. Worker Thread는 다시 다음 작업을 기다린다.

---

## 11.4 Worker Thread 수

기본값:

```text
4개
```

추후 확장:

```bash
./mini_db_server 8080 8
```

이런 식으로 Thread 수를 인자로 받을 수 있게 만들면 좋지만, 최소 구현에서는 고정값 4개로 충분하다.

## 12. Task Queue 명세

Task Queue는 클라이언트 요청을 Worker Thread에게 전달하기 위한 큐이다.

### 저장 데이터

```c
int client_fd;
```

최소 구현에서는 SQL 문자열을 Queue에 저장하지 않고, 연결된 소켓 FD만 저장한다.

이유:

* Main Thread는 요청 내용을 읽지 않는다.
* Worker Thread가 직접 요청을 읽고 처리한다.
* 구현이 단순하다.

---

### Queue 동기화

Task Queue는 여러 스레드가 동시에 접근하므로 아래 동기화 도구를 사용한다.

```c
pthread_mutex_t queue_mutex;
pthread_cond_t queue_cond;
```

## 14. 동시성 제어 명세

## 14.1 최소 구현 전략

최소 구현에서는 DB 전체에 하나의 Global Mutex를 둔다.

```c
pthread_mutex_t db_mutex;
```

SQL 실행 시 다음과 같이 처리한다.

```text
Worker Thread
  |
  v
db_mutex lock
  |
  v
db_execute(sql)
  |
  v
db_mutex unlock
```

---

## 14.2 이유

B+ Tree와 Storage는 여러 스레드가 동시에 접근하면 깨질 수 있다.

특히 다음 상황에서 문제가 생길 수 있다.

```text
Thread A: INSERT id=1
Thread B: SELECT id=1
Thread C: INSERT id=2
```

B+ Tree 구조가 수정되는 중에 다른 스레드가 접근하면 인덱스가 깨질 수 있다.

따라서 최소 구현에서는 모든 DB 작업을 하나의 Lock으로 보호한다.

---

## 14.3 장점

* 구현이 쉽다.
* 안정성이 높다.
* 발표 때 설명하기 좋다.
* 동시성 이슈를 고려했다는 점을 보여줄 수 있다.

---

## 14.4 단점

* 요청은 여러 Worker Thread가 받지만, 실제 DB 실행은 한 번에 하나씩 처리된다.
* 성능 최적화는 부족하다.

하지만 최소 구현에서는 이 방식이 가장 안전하다.

## 20. 발표용 핵심 설명

### 20.1 프로젝트 한 줄 소개

> MiniDB API Server는 기존 SQL 처리기와 B+ Tree 인덱스를 활용하여, 외부 클라이언트가 HTTP API를 통해 DBMS 기능을 사용할 수 있도록 만든 C 기반 미니 DBMS 서버입니다.

---

### 20.2 아키텍처 설명

> 클라이언트가 HTTP 요청으로 SQL을 보내면, 서버는 해당 연결을 Task Queue에 넣습니다. Worker Thread는 Queue에서 요청을 가져와 SQL을 실행하고, DB Engine은 기존 SQL 처리기와 B+ Tree 인덱스를 사용해 결과를 반환합니다.

---

### 20.3 Thread Pool 설명

> 요청마다 스레드를 새로 생성하지 않고, 서버 시작 시 미리 생성한 Worker Thread들이 요청을 처리하도록 구성했습니다. 이를 통해 여러 클라이언트 요청을 안정적으로 받을 수 있습니다.

---

### 20.4 동시성 제어 설명

> 여러 Worker Thread가 동시에 DB Engine과 B+ Tree에 접근하면 Race Condition이 발생할 수 있으므로, 최소 구현에서는 Global Mutex를 사용하여 DB 작업을 한 번에 하나씩 실행하도록 보호했습니다.

---

### 20.5 B+ Tree 설명

> `id` 기반 조회 시 전체 데이터를 순차 탐색하지 않고, 기존에 구현한 B+ Tree 인덱스를 통해 빠르게 레코드를 찾도록 구성했습니다.

## 23. 선택 구현

최소 구현이 끝난 뒤 시간이 남으면 아래 기능 중 하나만 추가한다.

| 기능              | 발표 효과                  | 난이도   |
| --------------- | ---------------------- | ----- |
| GET /stats      | Thread Pool 상태 보여주기 좋음 | 낮음    |
| 요청 로그           | 데모 보기 좋음               | 낮음    |
| JSON 요청 지원      | API 완성도 증가             | 중간    |
| Read/Write Lock | 동시성 설계 어필              | 중간    |
| TCP 모드 추가       | 차별화 가능                 | 중간 이상 |

가장 추천하는 선택 구현은 다음이다.

```text
GET /stats
```

이유는 구현은 쉽고, 발표 효과는 좋기 때문이다.

예시:

```json
{
  "status": "success",
  "worker_threads": 4,
  "total_requests": 25,
  "success_requests": 23,
  "failed_requests": 2
}
```

# 최종 최소 명세 요약

```text
프로젝트명: MiniDB API Server

구현 언어: C

서버 방식: HTTP

핵심 API:
- GET /health
- POST /query

요청 방식:
- POST /query body에 SQL 문자열 전달

지원 SQL:
- INSERT INTO users VALUES (...)
- SELECT * FROM users
- SELECT * FROM users WHERE id = ...

내부 구조:
- API Server
- Task Queue
- Thread Pool
- DB Engine
- SQL Processor
- B+ Tree Index

동시성 제어:
- Task Queue는 Mutex + Condition Variable
- DB Engine은 Global Mutex

테스트:
- curl 기반 API 테스트
- 동시 요청 테스트

발표 핵심:
- HTTP API 서버 구현
- Thread Pool 요청 처리
- 기존 SQL 처리기와 B+ Tree 연동
- Mutex를 통한 동시성 제어
```

이 명세대로 가면 범위가 작아서 하루 안에 완성 가능성이 높고, 발표 때도 “API 서버 + Thread Pool + DB 엔진 연동”이라는 과제 핵심을 충분히 보여줄 수 있다.
