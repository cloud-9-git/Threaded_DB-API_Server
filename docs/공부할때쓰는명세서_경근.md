# Mini DBMS API Server 사람용 학습 명세서

## 0. 이 문서의 목적

이 문서는 아래 질문에 답하기 위한 문서다.

```text

Thread Pool은 왜 필요한지?
rwlock은 왜 쓰는지?
각 파일과 함수는 무슨 역할인지?

```

즉, 이 문서는 **구현 지시서**라기보다 **이해용 지도**다.

---

# 1. 프로젝트를 한 문장으로 설명하면

> 기존에 만든 C 기반 미니 SQL 처리기를 HTTP API 서버로 감싸서, 외부 클라이언트가 SQL을 보내면 서버가 DB 작업을 수행하고 JSON으로 결과를 돌려주는 프로젝트다.

---

# 2. 현재 기존 프로젝트는 무엇인가?

현재 프로젝트의 핵심은 아직 API 서버가 아니다.
기존 프로젝트는 C로 작성된 **파일 기반 미니 SQL 처리기**다.

기존 실행 파일은 다음이다.

```bash
./sql_processor -d db -f queries/multi_statements.sql
```

기존 프로그램은 SQL 파일을 읽고 다음 흐름으로 실행된다.

```text
SQL 파일
  ↓
lexer
  ↓
parser
  ↓
executor
  ↓
schema / storage / B+Tree
  ↓
결과 출력
```

기존 프로젝트는 `.schema` 파일로 테이블 컬럼을 읽고, `.data` 파일에 row를 저장하며, `id` 컬럼이 있으면 in-memory B+Tree 인덱스를 만들어 `WHERE id = ...` 조회를 빠르게 처리한다. 기존 상태에서는 HTTP 서버, Thread Pool, JSON API, 동시성 제어는 구현되어 있지 않다. 

---

# 3. 이번 과제에서 새로 만드는 것

이번 과제에서는 기존 `sql_processor`를 없애는 게 아니다.

기존 DB 엔진은 그대로 두고, 그 앞에 **API 서버 계층**을 새로 붙인다.

새 실행 파일은 다음과 같다.

```bash
./mini_db_server -d db -p 8080 -t 4
```

새 구조는 이렇게 된다.

```text
Client
  ↓ HTTP 요청
mini_db_server
  ↓
Thread Pool
  ↓
DB API Adapter
  ↓
기존 SQL 처리기
  ↓
Storage + B+Tree
  ↓
JSON 응답
```

## 5.7 Thread

Thread는 하나의 프로그램 안에서 동시에 일하는 작업자다.

예를 들어 서버에 요청이 4개 들어왔을 때:

```text
Thread 1 → 요청 A 처리
Thread 2 → 요청 B 처리
Thread 3 → 요청 C 처리
Thread 4 → 요청 D 처리
```

이렇게 여러 요청을 동시에 처리할 수 있다.

---

## 5.8 Thread Pool

Thread Pool은 worker thread를 미리 만들어두는 구조다.

요청이 올 때마다 thread를 새로 만들면 느리고 위험하다.

그래서 서버 시작 시 미리 thread를 만들어둔다.

```text
서버 시작
  ↓
Worker Thread 4개 생성
  ↓
요청이 오면 놀고 있는 worker가 처리
```

---

## 5.9 Queue

Queue는 줄 서기 구조다.

먼저 들어온 요청이 먼저 처리된다.

```text
요청 A → 요청 B → 요청 C
```

서버에서는 main thread가 요청을 받아 queue에 넣고, worker thread들이 queue에서 꺼내 처리한다.

---

## 5.10 Bounded Queue

Bounded Queue는 크기 제한이 있는 queue다.

예:

```text
최대 1024개 요청까지만 대기 가능
```

왜 필요할까?

요청이 너무 많이 오면 무제한으로 쌓다가 메모리가 터질 수 있다.

그래서 제한을 둔다.

```text
queue 여유 있음 → 요청 저장
queue 가득 참 → 503 Service Unavailable 반환
```

---

## 5.11 Mutex

Mutex는 한 번에 하나만 들어갈 수 있게 막는 자물쇠다.

```text
Thread A lock 획득
Thread B 대기
Thread A unlock
Thread B 진입
```

안전하지만, 모든 요청이 하나씩만 실행되므로 성능이 낮아질 수 있다.

---

## 5.12 rwlock

rwlock은 Read-Write Lock이다.

규칙은 간단하다.

```text
읽기 SELECT는 여러 개 동시에 가능
쓰기 INSERT는 하나만 가능
```

즉:

```text
SELECT + SELECT + SELECT → 동시에 가능
INSERT → 혼자만 가능
INSERT 중 SELECT → SELECT 대기
SELECT 중 INSERT → INSERT 대기
```

이번 프로젝트에서는 `pthread_rwlock_t`를 사용한다.

---

# 6. 왜 무잠금 SELECT는 안 되는가?

처음에는 이렇게 생각할 수 있다.

```text
SELECT는 읽기만 하니까 lock 없이 해도 되지 않나?
```

하지만 이 프로젝트에서는 위험하다.

이유는 SELECT가 읽는 대상이 완전히 고정된 데이터가 아니기 때문이다.

SELECT는 다음을 읽을 수 있다.

```text
ExecutionContext
TableRuntime
B+Tree
.data 파일
```

그런데 INSERT는 동시에 다음을 바꾼다.

```text
.data 파일 append
next_id 증가
B+Tree insert
B+Tree split
runtime cache 변경
```

즉 SELECT가 lock 없이 실행되면, INSERT가 B+Tree를 수정하는 중간 상태를 읽을 수 있다.

예:

```text
Thread A: INSERT 중 B+Tree leaf split
Thread B: SELECT 중 B+Tree search
```

이때 Thread B가 깨진 중간 포인터를 보면 잘못된 결과나 crash가 날 수 있다.

그래서 이번 프로젝트에서는:

```text
SELECT → read lock
INSERT → write lock
```

을 사용한다.

# 7. 이번 프로젝트의 동시성 정책

## 7.1 기본 정책

```text
SELECT만 있는 요청:
  read lock 사용

INSERT가 하나라도 있는 요청:
  write lock 사용
```

---

## 7.2 SELECT-only 요청

예:

```sql
SELECT * FROM users WHERE id = 1;
```

이 요청은 read lock으로 실행한다.

하지만 바로 read lock을 잡으면 안 된다.

왜냐하면 SELECT도 처음 테이블에 접근하면 runtime cache를 만들 수 있기 때문이다.

그래서 순서는 이렇게 한다.

```text
1. SQL을 먼저 분석한다.
2. SELECT에서 사용하는 table 이름을 찾는다.
3. write lock을 잠깐 잡고 table runtime을 미리 로드한다.
4. write lock을 푼다.
5. read lock을 잡고 SELECT를 실행한다.
6. read lock을 푼다.
```

쉽게 말하면:

```text
준비 작업은 write lock
진짜 조회는 read lock
```

---

## 7.3 INSERT 포함 요청

예:

```sql
INSERT INTO users VALUES ('kim', 25);
```

또는:

```sql
INSERT INTO users VALUES ('kim', 25);
SELECT * FROM users WHERE id = 1;
```

INSERT가 하나라도 있으면 전체를 write lock으로 실행한다.

이유:

```text
1. INSERT는 파일을 수정한다.
2. INSERT는 B+Tree를 수정한다.
3. INSERT는 next_id를 수정한다.
4. 같은 요청 안의 SELECT는 방금 INSERT한 결과를 봐야 한다.
```

# 9. 요청 하나가 처리되는 전체 흐름

예를 들어 클라이언트가 이 요청을 보낸다고 하자.

```bash
curl -X POST http://localhost:8080/query \
  -H "Content-Type: application/json" \
  -d '{"sql":"SELECT * FROM users WHERE id = 1;"}'
```

서버 내부에서는 다음 순서로 움직인다.

```text
1. Client가 TCP 연결을 건다.
2. Server의 accept()가 client_fd를 만든다.
3. Main Thread가 client_fd를 Task Queue에 넣는다.
4. Worker Thread가 Task Queue에서 client_fd를 꺼낸다.
5. Worker Thread가 HTTP request를 읽는다.
6. HTTP parser가 method, path, body를 분리한다.
7. JSON parser가 body에서 sql 문자열을 꺼낸다.
8. DbApi가 SQL을 실행한다.
9. 기존 lexer/parser/executor가 실행된다.
10. ExecResult가 만들어진다.
11. JSON writer가 ExecResult를 JSON으로 바꾼다.
12. HTTP response를 client_fd로 보낸다.
13. client_fd를 닫는다.
```

그림으로 보면:

```text
curl
  ↓
accept()
  ↓
TaskQueue
  ↓
Worker Thread
  ↓
http_read_request()
  ↓
json_extract_string_field()
  ↓
db_api_execute_sql()
  ↓
tokenize_sql()
  ↓
parse_next_statement()
  ↓
execute_statement()
  ↓
json response
```

## 10.7 `thread_pool.c`

# 11. 주요 구조체 쉽게 이해하기

## 11.1 `ServerConfig`

서버 실행 설정이다.

```c
typedef struct {
    const char *db_dir;
    int port;
    size_t thread_count;
} ServerConfig;
```

뜻:

```text
db_dir       어느 DB 폴더를 사용할지
port         몇 번 포트에서 기다릴지
thread_count worker thread를 몇 개 만들지
```

---

## 11.2 `Server`

서버 전체 상태다.

```c
typedef struct {
    ServerConfig config;
    int listen_fd;
    ThreadPool pool;
    DbApi db_api;
    ServerStats stats;
} Server;
```

뜻:

```text
config     설정
listen_fd  클라이언트 연결을 기다리는 소켓
pool       thread pool
db_api     DB 엔진 연결부
stats      요청 통계
```

---

## 11.3 `Task`

worker가 처리할 일이다.

```c
typedef struct {
    int client_fd;
} Task;
```

뜻:

```text
client_fd = 클라이언트와 연결된 소켓 번호
```

이번 프로젝트에서는 task에 SQL 문자열을 넣지 않는다.

왜냐하면 Main Thread는 요청 내용을 읽지 않고, Worker Thread가 직접 읽기 때문이다.

# 13. 기존 SQL 엔진 이해하기

```text

```

## 13.1 Lexer

문자열 SQL을 token으로 나눈다.

입력:

```sql

```

출력 느낌:

```text

FROM

WHERE

=
1
;

```

## 13.2 Parser

token들을 보고 문장 구조를 만든다.

예:

```sql

```


```text

```

# 15. B+Tree는 왜 쓰는가?

전체 데이터를 하나씩 찾으면 느리다.

예를 들어 users.data에 100만 줄이 있다고 하자.

```sql
SELECT * FROM users WHERE id = 777777;
```

인덱스가 없으면:

```text
1번 row 확인
2번 row 확인
3번 row 확인
...
777777번 row 확인
```

이렇게 오래 걸린다.

B+Tree가 있으면:

```text
id 777777이 있는 row_offset을 빠르게 찾음
그 offset으로 파일에서 바로 읽음
```

이번 프로젝트에서는 `WHERE id = ...` 조회가 B+Tree를 사용할 수 있다.

응답에는 이 정보가 들어간다.

```json
"used_index": true
```

# 20. 발표용 쉬운 설명

발표 때 이렇게 말하면 된다.

> 저희 프로젝트는 기존 CLI 기반 SQL 처리기를 HTTP API 서버로 확장한 미니 DBMS입니다.
> 클라이언트는 `POST /query`로 SQL을 JSON 형태로 보내고, 서버는 Thread Pool의 Worker Thread에게 요청을 분배합니다.
> Worker Thread는 HTTP와 JSON을 파싱한 뒤 기존 lexer, parser, executor를 호출해 SQL을 실행하고, 결과를 JSON으로 반환합니다.
> 동시성 제어는 `pthread_rwlock_t`를 사용했습니다. SELECT-only 요청은 read lock으로 처리해 여러 조회가 동시에 가능하게 했고, INSERT가 포함된 요청은 write lock으로 처리해 `.data` 파일 append, auto-id 증가, B+Tree 수정 과정에서 race condition이 발생하지 않도록 했습니다.

# 23. 이 프로젝트의 한계

현재 목표는 하루 안에 완성 가능한 API 서버다.

그래서 일부 기능은 의도적으로 제외한다.

구현하지 않는 것:

```text
CREATE TABLE
UPDATE
DELETE
JOIN
ORDER BY
트랜잭션
rollback
인증/인가
영구 저장되는 B+Tree 인덱스
완전한 HTTP parser
완전한 JSON parser
```

특히 transaction이 없기 때문에 이런 일이 가능하다.

```sql
INSERT INTO users VALUES ('kim', 25);
SELCT * FROM users;
```

첫 번째 INSERT는 성공하고, 두 번째 SQL에서 에러가 나도 INSERT는 rollback되지 않는다.

이건 기존 CLI 동작과 맞춘 것이다.

# 24. 개선 방향

시간이 남으면 다음을 개선할 수 있다.

## 24.1 table 단위 lock

지금은 DB 전체에 하나의 rwlock을 둔다.

나중에는 테이블마다 lock을 둘 수 있다.

```text
users table lock
products table lock
orders table lock
```

이렇게 하면 서로 다른 테이블 작업은 더 많이 병렬 처리할 수 있다.

---

## 24.2 더 완전한 JSON parser

현재는 `"sql"` 문자열만 꺼내면 된다.

나중에는 다음도 지원할 수 있다.

```json
{
  "sql": "...",
  "timeout_ms": 1000,
  "readonly": true
}
```

---

## 24.3 persistent index

현재 B+Tree는 메모리에만 있다.

서버가 꺼지면 사라지고, 다시 시작할 때 `.data`에서 다시 만든다.

나중에는 index 파일을 따로 저장할 수 있다.

---

## 24.4 DELETE / UPDATE

현재는 INSERT와 SELECT만 지원한다.

나중에는:

```sql
UPDATE users SET age = 30 WHERE id = 1;
DELETE FROM users WHERE id = 1;
```

을 추가할 수 있다.

하지만 이 경우 B+Tree와 파일 포맷 설계를 다시 해야 한다.

# 25. 최종 요약

이 프로젝트는 크게 네 덩어리로 이해하면 된다.

```text
1. HTTP 서버
2. Thread Pool
3. DB API Adapter
4. 기존 SQL Processor
```

요청 흐름은 다음 한 줄로 기억하면 된다.

```text
HTTP 요청 → Worker Thread → JSON에서 SQL 추출 → 기존 DB 엔진 실행 → JSON 응답
```

동시성 정책은 다음 한 줄로 기억하면 된다.

```text
SELECT는 read lock, INSERT는 write lock, SELECT 전에는 runtime preload
```

발표 핵심은 다음 세 가지다.

```text
1. 기존 SQL 처리기를 API 서버로 확장했다.
2. Thread Pool로 여러 요청을 처리한다.
3. rwlock으로 SELECT 병렬성과 INSERT 안전성을 균형 있게 잡았다.
```
